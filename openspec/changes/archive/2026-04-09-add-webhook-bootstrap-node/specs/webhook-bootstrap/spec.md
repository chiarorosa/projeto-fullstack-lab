## ADDED Requirements

### Requirement: Webhook Node Defines Bootstrap Endpoint
The system SHALL allow a flow to declare a Webhook Node as a bootstrap source with a stable, unique trigger path.

#### Scenario: Webhook bootstrap node configured
- **WHEN** a user adds and configures a Webhook Node in a flow
- **THEN** the system stores a unique webhook trigger identifier bound to that flow and node
- **AND** the identifier remains stable across executions until the node is deleted or regenerated by explicit user action

#### Scenario: Duplicate webhook path conflict
- **WHEN** a webhook identifier collision is detected during node creation or regeneration
- **THEN** the system rejects the conflicting identifier
- **AND** generates or requests a non-conflicting identifier before activation

### Requirement: Webhook Trigger Starts Flow Execution
The system SHALL accept inbound webhook requests and start execution for the mapped flow using the Webhook Node as bootstrap input.

#### Scenario: Valid webhook request received
- **WHEN** a request is received on a registered webhook trigger path
- **THEN** backend resolves the associated flow and bootstrap node
- **AND** starts a new run using request payload and metadata as execution input context

#### Scenario: Unknown webhook path called
- **WHEN** a request is received for a webhook trigger path that is not registered
- **THEN** the system returns a not-found response
- **AND** no flow execution is started

### Requirement: Webhook Request Validation and Response Contract
The system SHALL validate webhook requests and return explicit response outcomes for accepted and rejected triggers.

#### Scenario: Request fails validation
- **WHEN** a webhook request is malformed or missing required bootstrap fields
- **THEN** the system returns a validation error response describing the failure reason
- **AND** does not start execution

#### Scenario: Request accepted for asynchronous run
- **WHEN** a webhook request passes validation and execution is queued or started
- **THEN** the system returns an accepted response with run reference metadata
- **AND** downstream execution proceeds independently of the HTTP response lifecycle

### Requirement: Webhook Run Traceability
The system SHALL persist webhook-origin metadata in run artifacts for auditing and debugging.

#### Scenario: Webhook-triggered run is persisted
- **WHEN** a run starts from webhook bootstrap
- **THEN** run metadata includes trigger source `webhook`, trigger identifier, request timestamp, and request correlation identifier
- **AND** artifact views can distinguish webhook-triggered runs from Task Node-triggered runs
